Migrer Supabase → OVH VPS ?

Comparaison de l'architecture actuelle vs migration complète vs approche progressive

Situation actuelle — problèmes connus

🔴 VITE_SUPABASE_SERVICE_KEY exposée dans le bundle JS → n'importe qui peut lire/modifier la DB
🟡 Dépendance externe Supabase (coût, RGPD, disponibilité)
🟡 Supabase Auth intégré partout (login, reset password, sessions)
🟡 Fichiers partitions stockés dans Supabase Storage

Option A — Garder Supabase

Node.js/Hono gère uniquement les appels IA.
Le frontend appelle toujours Supabase directement.
La service key reste exposée côté frontend.

Pour

  • Rapide à implémenter
  • Rien à migrer

Contre

  • Service key toujours exposée
  • Problème sécurité non résolu

Option B — Migration progressive Recommandée

Node.js/Hono devient l'unique point d'entrée du frontend.
Il appelle Supabase côté serveur (service key cachée).
Plus tard, on remplace Supabase par PostgreSQL self-hosted sans toucher au frontend.

Frontend → Hono API → Supabase (phase 1)
Frontend → Hono API → PostgreSQL OVH (phase 2, plus tard)

Pour

  • Service key sécurisée dès phase 1
  • Migration Supabase optionnelle, sans refonte frontend
  • RGPD : données restent EU

Contre

  • Réécrire tous les appels frontend (Supabase client → fetch Hono)
  • Auth à migrer (Supabase Auth → JWT custom ou better-auth)

Option C — Migration complète maintenant

PostgreSQL self-hosted sur OVH VPS (via Coolify).
Auth custom (better-auth ou Lucia).
Stockage fichiers : Minio sur VPS.
Frontend appelle uniquement l'API Hono.

Pour

  • Autonomie totale
  • Zéro dépendance externe
  • Coût minimal

Contre

  • Gros chantier (3-4 semaines)
  • Backups, maintenance DB à gérer soi-même
  • Migration données à orchestrer

Ce que "Migration progressive" change pour le module IA

✅ Les outils IA n'appellent jamais Supabase directement
✅ Ils appellent des endpoints Hono internes : GET /api/members, GET /api/planning/:year...
✅ Quand on migre la DB plus tard, seul Hono change — le module IA reste intact
✅ La clé Anthropic est sur Hono, la service key Supabase est sur Hono → 0 secret dans le frontend
